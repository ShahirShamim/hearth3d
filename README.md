# Hearth3D 🏠

A real-time **isometric dashboard for Home Assistant**, packaged as a single self-contained Docker container.

Hearth3D connects to your local Home Assistant instance, automatically discovers your **areas, devices, and entities**, and renders them as a dark-mode isometric floor plan. Rooms become grid cards; lights, switches, media players and more appear on the room they belong to. States update live, and clicking a device toggles it in Home Assistant.

```
Browser ◄── WebSocket (port 8080) ──► Node.js backend ◄── HA WebSocket API ──► Home Assistant
   ▲                                        │
   └───────── static React frontend ◄──────┘
```

Everything is generated by one script: **`setup.sh`** writes out the complete project (backend, frontend, Dockerfile, compose file) with no placeholders — run it, add your token, and you're live.

## Features

- 🔍 **Zero configuration mapping** — rooms and device placement come straight from your Home Assistant area registry. Entity-level area overrides are respected; disabled/hidden entities are skipped.
- ⚡ **Real-time** — subscribes to `state_changed` events and streams updates to the browser instantly. Renaming areas or reassigning devices in HA updates the dashboard without a restart.
- 💡 **Click to control** — toggle lights, switches, fans, media players, covers, and locks from the floor plan.
- ✏️ **Edit mode** — flattens the view to top-down so you can drag devices anywhere on their room card. Layout persists in your browser (local storage), with one-click reset.
- 🔒 **Token stays server-side** — the browser never sees your Home Assistant credentials, and clients can only send toggle commands for entities the dashboard exposes.
- 🐳 **Single container** — multi-stage Docker build; the final image serves both the API bridge and the static frontend on one port.

## Requirements

- Docker with the Compose plugin (`docker compose`) — or `docker-compose`
- A running Home Assistant instance reachable from the machine running Docker
- A Home Assistant **Long-Lived Access Token**

## Quick start

**1. Generate the project**

```sh
git clone https://github.com/ShahirShamim/hearth3d.git
cd hearth3d
./setup.sh
cd hearth3d
```

**2. Create a Home Assistant access token**

In Home Assistant: click your profile (bottom-left avatar) → **Security** → **Long-lived access tokens** → **Create token**. Copy it — it is shown only once.

**3. Configure the environment**

```sh
cp .env.example .env
```

Edit `.env`:

```ini
HA_URL=http://192.168.1.50:8123   # your Home Assistant IP address
HA_TOKEN=eyJhbGciOi...            # the token you just created
```

> ⚠️ **Use the IP address of your Home Assistant host, not `homeassistant.local`.**
> mDNS names usually do not resolve inside Docker containers. On Linux you can
> alternatively uncomment `network_mode: host` in `docker-compose.yml`.

**4. Build and run**

```sh
docker compose up --build
```

**5. Open the dashboard**

Visit <http://localhost:8080> — or `http://<docker-host-ip>:8080` from any device on your LAN.

## Using the dashboard

| Action | How |
|---|---|
| Toggle a device | Click its icon (lights glow when on) |
| Rearrange devices | **Edit layout** in the header → drag devices → **Done editing** |
| Reset positions | **Reset layout** (visible while editing) |
| Check backend status | Header badge shows Home Assistant connection state; `GET /health` returns JSON |

## Configuration

All configuration is via environment variables (set them in `.env`):

| Variable | Default | Description |
|---|---|---|
| `HA_URL` | `http://homeassistant.local:8123` | Base URL of your Home Assistant instance (use its IP) |
| `HA_TOKEN` | *(required)* | Long-lived access token |
| `PORT` | `8080` | Dashboard HTTP/WebSocket port |
| `DOMAINS` | `light,switch,media_player,fan,cover,lock,climate,vacuum` | Entity domains shown on the dashboard |

Want temperature sensors on the floor plan? Add `sensor` to `DOMAINS`.

## Local development (without Docker)

```sh
# Terminal 1 — backend
cd hearth3d/server && npm install
HA_URL=http://192.168.1.50:8123 HA_TOKEN=... node server.js

# Terminal 2 — frontend with hot reload (proxies /ws to the backend)
cd hearth3d/frontend && npm install
npm run dev
```

## Troubleshooting

| Symptom | Cause / fix |
|---|---|
| `auth_invalid` in container logs | Token is wrong or revoked — create a new one |
| `HA connection closed; retrying…` forever | Container can't reach `HA_URL`. Use the IP address, verify port 8123, check firewalls |
| Dashboard is empty | Your entities aren't assigned to areas. In HA: Settings → Devices & services → Devices → assign areas |
| A device is missing | Its domain isn't in `DOMAINS`, or the entity is disabled/hidden in HA |
| `Set HA_TOKEN in a .env file` error on startup | Compose fails fast when the token is missing — create `.env` from `.env.example` |

## Security notes

- The dashboard has **no authentication of its own** — anyone on your LAN who can reach port 8080 can view and toggle the exposed devices. Do not port-forward it to the internet; put it behind a reverse proxy with authentication for remote access.
- The HA token never leaves the backend, and browser clients cannot invoke arbitrary Home Assistant services — only `toggle` on displayed entities.
